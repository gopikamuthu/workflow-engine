What's Inside

Everything You Need

Built with Node.js + Express backend, MongoDB database, and a fully interactive frontend dashboard.

⬡
Workflow Builder
Create multi-step workflows with a visual editor. Define input schemas, configure steps, and set up routing rules — all from the dashboard.
⚡
Rule Engine
Powerful condition evaluator supporting &&, ||, comparisons, and DEFAULT fallbacks. Priority-ordered evaluation for precise routing.
▶
Live Execution
Watch workflows execute in real-time with animated step transitions. See rule evaluations, matched conditions, and next-step routing as they happen.
🗄️
MongoDB Storage
All workflows and executions are stored in MongoDB. Data persists across server restarts — no more losing your work.
📋
Audit Log
Complete execution history for compliance and tracking. Every rule evaluation, step transition, and approver action is recorded with timestamps.
⟳
Retry & Recovery
Failed executions can be retried from the failed step only — not the entire workflow. Version control ensures reproducibility across retries.
How It Works

From Creation to Execution

Set up a workflow in minutes and start automating your processes.

1
Create a Workflow
Define the workflow name, input schema (fields, types, allowed values), and configure your steps — Task, Approval, or Notification.
POST /api/workflows → { name: "Expense Approval", steps: [...] }
2
Add Rules to Steps
Define conditions for each step. Rules are evaluated by priority order — the first matching rule decides the next step to execute.
{ condition: 'amount > 100 && priority == "High"', next_step_id: "ceo-approval" }
3
Execute with Input Data
Trigger a workflow execution by submitting input data. The rule engine evaluates conditions at each step and routes automatically.
POST /api/workflows/wf-001/execute → { data: { amount: 250, country: "US" } }
4
Track & Audit
View execution logs showing every rule evaluated, step completed, and decision made. Retry failed steps without restarting the entire workflow.
GET /api/executions/:id → { logs: [...], status: "completed" }
